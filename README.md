#  «12.8. Резервное копирование баз данных — Сараев Антон».
## Задание 1
Варианты резервного копирования для различных сценариев
####  1.1. Восстановление данных за предыдущий день
Для этого случая оптимально подходит ежедневное полное резервное копирование. Каждый день создаётся полная копия базы данных, что позволяет при необходимости восстановить все данные на состояние конца предыдущего дня. Такой подход обеспечивает максимальную надёжность и простоту восстановления, хотя требует больше места для хранения и времени на выполнение бэкапа.

####  1.2. Восстановление данных за час до предполагаемой поломки
Здесь рекомендуется использовать комбинированную схему: ежедневное полное резервное копирование в сочетании с инкрементальными или дифференциальными копиями, а также журналирование транзакций (transaction log backup). Например, ночью выполняется полный бэкап, а в течение дня — инкрементальные копии (например, каждый час). Это позволяет восстановить базу на любой момент времени в пределах суток, минимизируя потери данных.

####  1.3. Моментальное переключение при поломке базы
В этом случае используется резервный сервер (standby), который постоянно синхронизируется с основным. При сбое основного сервера происходит автоматическое переключение на резервный, что обеспечивает минимальное время простоя и практически мгновенное восстановление работы. Для этого применяются технологии такие как Always On Availability Groups в MS SQL Server, Oracle Data Guard, MySQL Replication и другие.

## Задание 2
#### 2.1. Пример команд для резервирования и восстановления
Для создания резервной копии базы данных PostgreSQL используется утилита pg_dump. Пример команды для создания дампа базы mydatabase от имени пользователя myuser:
```bash
pg_dump -U myuser -F c -f mydatabase_backup.dump mydatabase
```
* -U myuser — имя пользователя.
* -F c — формат дампа (custom, рекомендуется для гибкости).
* -f mydatabase_backup.dump — имя выходного файла.
* mydatabase — имя базы данных.

Для восстановления базы из резервной копии используется pg_restore:
```bash
pg_restore -U myuser -d mydatabase -C mydatabase_backup.dump
```
* -d mydatabase — имя целевой базы.
* -C — создать базу данных перед восстановлением (если её нет).
#### 2.1.*Автоматизация процесса
Да, автоматизация возможна и широко применяется на практике. Для этого:

* создаются скрипты (например, на bash или Python), которые выполняют команды pg_dump/pg_restore по расписанию;
* используется cron (в Linux) для запуска скриптов в заданное время (например, каждую ночь);
* для более сложных сценариев применяются системы управления задачами, такие как pgBackRest, Barman или Ansible, которые позволяют настраивать расписания, ротацию бэкапов, уведомления и проверку целостности копий.
## Задаине 3
#### 3.1. Пример команды инкрементного резервного копирования
Для инкрементного резервного копирования в MySQL используется утилита Percona XtraBackup
```sql
xtrabackup --backup --target-dir=/path/to/inc_backup \
    --incremental-basedir=/path/to/full_backup
```
* --backup — указывает, что выполняется резервное копирование.
* --target-dir — директория для хранения инкрементной копии.
* --incremental-basedir — путь к предыдущей полной или инкрементной копии, относительно которой вычисляются изменения.
После создания инкрементных копий их можно применить к исходной полной резервной копии для восстановления актуального состояния базы.

#### 3.1.*Преимущества использования реплики
Использование реплики (slave/replica) для резервного копирования даёт ряд преимуществ по сравнению с обычным бэкапом основной базы:

* Минимизация нагрузки на основную базу: резервное копирование выполняется на реплике, что не влияет на производительность и доступность основной базы данных.
* Возможность создания бэкапов без остановки работы: реплика может быть временно остановлена или заморожена для снятия консистентной копии, не прерывая работу приложения.
* Снижение времени простоя: в случае сбоя основной базы реплика может быстро взять на себя её функции (failover), обеспечивая высокую доступность.
* Тестирование и разработка: на реплике можно безопасно тестировать восстановление, анализировать данные или выполнять ресурсоёмкие запросы, не рискуя основной базой.
## Итог
Инкрементное резервное копирование с помощью XtraBackup позволяет эффективно сохранять только изменённые данные, а использование реплики — снизить риски и нагрузку на продуктивную среду, что особенно важно для крупных и нагруженных систем.
